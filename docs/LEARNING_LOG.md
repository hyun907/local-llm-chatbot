## STEP 0. Ollama와 Apple Silicon 메모리 구조

- **Ollama**: 로컬에서 LLM을 실행하는 런타임. 내부적으로 llama.cpp 기반이며,
  모델 다운로드(레지스트리) + 실행(추론 서버) + REST API 제공을 한 번에 해준다.
  기본 포트는 `11434`.
- **Apple Silicon 통합메모리(Unified Memory)**: CPU와 GPU가 하나의 메모리 풀을
  공유한다. "GPU VRAM 16GB"가 아니라 OS·앱·모델이 16GB를 나눠 쓰는 구조라,
  모델 크기 선택 시 dGPU VRAM 8GB급으로 보수적으로 잡아야 스왑을 피할 수 있다.
- **서버 기동 옵션** (Homebrew caveat 권장값):
  - `OLLAMA_FLASH_ATTENTION=1`: 어텐션 연산의 메모리 효율 개선.
  - `OLLAMA_KV_CACHE_TYPE=q8_0`: KV 캐시(대화 컨텍스트 저장 공간)를 8bit로
    양자화해 메모리 절약. 긴 컨텍스트일수록 효과가 큼.

## STEP 1. 모델 태그와 양자화

- `ollama pull llama3.2`의 기본(latest) 태그는 **3B 파라미터, Q4_K_M 양자화**
  버전으로 2.0GB. 원본 FP16 3B는 약 6GB인데 4bit 양자화로 1/3 수준이 된 것.
- **양자화(Quantization)**: 가중치를 16bit → 4bit 등으로 낮춰 메모리·속도를
  얻고 약간의 품질을 내주는 트레이드오프. Q4_K_M은 "4bit + 중요 레이어는
  더 높은 정밀도"를 쓰는 혼합 방식.
- `latest` 태그는 시점에 따라 가리키는 내용이 바뀔 수 있으므로, 재현성이
  필요하면 `ollama list`의 **ID(digest)** 를 함께 기록해야 한다.
  (이번 실행: `llama3.2:latest` = `a80c4f17acd5`)

## STEP 2~3. Ollama의 세 가지 API

| API | 용도 | 요청 형식 |
|---|---|---|
| `/api/generate` | 단발 텍스트 완성 | `prompt` 문자열 하나 |
| `/api/chat` | 대화형 (멀티턴) | `messages: [{role, content}, ...]` 배열 |
| `/v1/chat/completions` | OpenAI 호환 | OpenAI SDK 그대로 사용 가능 |

- **스트리밍 방식**: `stream: true`면 응답이 NDJSON(한 줄에 JSON 하나)으로
  토큰 단위 도착한다. 각 청크의 `response`를 이어 붙이고 `done: true`에서
  종료. OpenAI의 SSE(`data: ...`) 형식과 다르다는 점에 주의.
- **성능 메트릭**: 논스트리밍 응답 마지막에 `eval_count`(생성 토큰 수)와
  `eval_duration`(나노초)이 붙는다. 토큰 속도 = `eval_count / eval_duration × 1e9`.
  이번 측정: 51.3 tok/s (llama3.2 3B, M5).
- **OpenAI 호환 레이어**: `base_url`만 `http://localhost:11434/v1`로 바꾸면
  기존 OpenAI SDK 코드가 그대로 동작한다. `api_key`는 SDK에서 필수라 아무
  값이나 넣어야 하지만 Ollama는 검증하지 않는다. → 기존 OpenAI 기반 코드를
  로컬 LLM으로 갈아끼울 때 코드 수정이 거의 없다는 게 핵심 이점.
- **작은 모델의 한국어 한계**: llama3.2 3B는 한국어 질문에 베트남어·중국어가
  섞인 답("Tôi là...", "我的")을 내는 **코드 스위칭** 현상을 보였다. 다국어
  학습 데이터에서 한국어 비중이 작은 소형 모델의 전형적 한계. 한국어 중심
  서비스라면 qwen2.5 등 한국어 비중이 높은 모델이 필요함을 시사한다.

## STEP 4~5. Flask 챗봇 설계에서 배운 것

- **대화 이력 무결성 패턴**: 요청마다 `chat_history + [새 user 메시지]`로
  **복사본**을 만들어 Ollama에 보내고, 응답이 성공했을 때만 user/assistant
  쌍을 전역 이력에 append 한다. 실패 시점에 이력이 절대 변하지 않으므로
  "실패한 질문이 다음 턴 컨텍스트에 유령처럼 남는" 버그를 원천 차단한다.
  (DB의 트랜잭션 커밋과 같은 발상)
- **에러의 HTTP 의미론**: 업스트림(Ollama) 장애를 클라이언트에 그대로 던지지
  않고 게이트웨이 관점의 상태 코드로 번역했다.
  - 연결 불가/연결 타임아웃 → **503** Service Unavailable (서버 켜면 해결)
  - 응답 대기 초과 → **504** Gateway Timeout
  - Ollama가 반환한 오류(모델 없음 등) → **502** Bad Gateway + 원인 메시지
  - 클라이언트 잘못(JSON 오류, 빈 메시지) → **400**
- **macOS 포트 함정**: AirPlay Receiver가 5000 포트에서 리슨하므로 Flask 기본
  포트 5000은 macOS에서 자주 충돌한다. `lsof -nP -iTCP:5000 -sTCP:LISTEN`으로
  확인 가능. 이 프로젝트는 5001을 기본값으로 사용.
- **Flask 세부**: `request.get_json(silent=True)`는 파싱 실패 시 예외 대신
  `None`을 반환하므로 400 처리를 직접 제어할 수 있다. `render_template_string`
  으로 단일 파일 안에 프론트엔드를 내장(실습 규모에서는 파일 분리보다 간단).
- **`ollama ps` 읽는 법**: 로드된 모델의 실제 상주 크기(가중치+KV캐시,
  llama3.2는 2.3GB), 처리 장치(100% GPU = Metal), 컨텍스트 길이(4096),
  자동 언로드까지 남은 시간(기본 keep_alive 5분)을 보여준다.

## STEP 6. 모델 비교에서 배운 것

- **LLM 추론은 메모리 대역폭에 묶인다**: 토큰 하나를 생성할 때마다 모델
  가중치 전체를 메모리에서 읽어야 하므로, 생성 속도 ≈ 대역폭 ÷ 모델 크기.
  실측에서 2GB급(3B Q4)이 ~51 tok/s, 4.4~4.7GB급(7B Q4)이 ~26 tok/s로
  크기에 거의 정확히 반비례했다. 파라미터 수가 아니라 **디스크 상 크기**가
  속도를 예측하는 지표.
- **cold vs warm의 실체**: 차이는 `load_duration`(가중치를 메모리에 올리는
  시간, 1~2초)이다. Ollama는 기본 5분(keep_alive) 후 모델을 내리므로, 첫
  요청만 느린 현상은 재로딩 때문. 벤치마크에서 cold를 따로 재려면
  `ollama stop <model>`로 명시적으로 언로드해야 한다.
- **체감 속도 ≠ tok/s**: qwen2.5와 mistral은 tok/s가 같지만(26), qwen2.5는
  간결하게 답해(100토큰 vs 310토큰) 총 응답 시간이 1/3이다. 사용자 체감은
  `총 시간 = 토큰 수 ÷ tok/s`라서, 장황한 모델은 빨라도 느리게 느껴진다.
- **다국어 품질은 크기 순이 아니다**: 3.8B phi3:mini(영어 특화)는 한국어가
  붕괴됐고, 같은 7B라도 mistral(유럽어 중심)은 유창하지만 환각이 심했으며,
  qwen2.5(CJK 데이터 풍부)만 정확했다. 모델 선택은 벤치마크 순위가 아니라
  **대상 언어의 학습 데이터 비중**을 봐야 한다.
- **temperature=0 + seed 고정**: 같은 조건에서 warm 실행 간 응답이 동일하고
  tok/s 편차가 ±0.4 이내 → 비교 실험의 재현성을 확보하는 기본기.

## 사후 개선. temperature와 글리치 토큰

- **temperature의 실체**: 다음 토큰의 확률 분포를 얼마나 "평평하게" 만들지의
  계수. 높을수록(0.8~) 저확률 토큰도 뽑힐 수 있어 다양하지만 위험하고,
  0이면 항상 최고 확률 토큰만 뽑아 결정적이지만 단조롭다.
- **글리치 토큰 누출**: 코드 데이터를 많이 학습한 모델(qwen2.5 등)은 어휘에
  `numel`, `endregion` 같은 프로그래밍 토큰이 많고, 문장 서두처럼 불확실성이
  큰 지점 + 높은 temperature 조합에서 이런 토큰이 간헐적으로 샘플링된다.
  일회성 노이즈이며 모델 자체 결함이 아니라 **샘플링 설정 문제**에 가깝다.
- **실전 절충값**: 챗봇 용도로 temperature 0.3~0.5가 "누출 억제 + 답변 다양성
  유지"의 절충 구간. 이 프로젝트는 0.4 적용 후 동일 질문 3회에서 누출 0회,
  답변은 서로 다름을 확인했다. 벤치마크/테스트는 0(재현성), 창작 용도는
  0.7 이상처럼 **용도별로 달리 잡는 것**이 핵심.

## 사후 개선. SSE 스트리밍에서 배운 것

- **SSE vs WebSocket**: 챗봇 응답처럼 서버→클라이언트 단방향 스트림은
  SSE(`text/event-stream`)가 적합. HTTP 하나로 끝나 프록시·인증도 그대로
  동작한다. WebSocket은 양방향이 필요할 때(협업 편집, 게임 등) 쓰는 도구.
- **형식 변환 체인**: Ollama는 NDJSON(한 줄에 JSON 하나), SSE는
  `data: {...}\n\n`(빈 줄로 이벤트 구분). Flask 제너레이터가 중간에서 형식만
  바꿔 중계하면 된다. 브라우저 쪽은 POST라서 `EventSource` 대신
  `fetch` + `ReadableStream`으로 직접 파싱했다.
- **Flask 스트리밍의 함정 — 요청 컨텍스트**: `Response(generator)`의 제너레이터
  본문은 뷰 함수가 반환된 **뒤에** 실행되므로 `session` 같은 요청 컨텍스트에
  접근할 수 없다. 세션 이력 리스트의 **참조를 클로저로 미리 캡처**해서 해결
  (Flask의 `stream_with_context`를 쓰는 방법도 있다).
- **스트리밍 + 이력 무결성**: "성공 후에만 커밋" 패턴을 스트리밍에 맞게
  재정의하면 "done 신호까지 완주한 경우에만 커밋"이 된다. 중간에 끊긴 절반짜리
  응답은 화면에는 보였더라도 이력에는 남지 않는다.
- **체감 지연의 분해**: 총 시간 = 첫 토큰까지(TTFT) + 생성 시간. 스트리밍은
  총 시간을 줄이는 게 아니라 **"빈 화면 시간"을 TTFT(0.25초)로 줄이는 것**.
  실측으로 첫 글자 표시가 ~3초 → 0.25초가 됐고, 이것이 같은 모델·같은
  속도에서 체감이 완전히 달라지는 이유다.

## 사후 개선 2. ":numel" 코드 토큰 누출 (temperature 조정)

**이슈**: 브라우저에서 "점심 추천해달라"고 하자 답변이 `:numel`로 시작함.

```
:numel
순대국밥을 추천드립니다. 맛있고 따뜻한 점심이 되셨으면 좋겠습니다!
```

**원인 분석**: `numel`은 PyTorch·MATLAB의 "배열 원소 개수" 함수명. qwen2.5는
코드 학습 데이터가 많아 어휘에 프로그래밍 토큰이 다수 포함돼 있는데, Ollama
기본 temperature(0.8)에서는 문장 서두처럼 다음 토큰 불확실성이 큰 지점에서
저확률 코드 토큰이 샘플링될 수 있다. 벤치마크의 phi3:mini ` endregion` 누출과
같은 종류이며, temperature=0으로 돌린 벤치마크에서는 발생하지 않았던 것과도
부합한다.

**해결**: `/chat`의 Ollama 호출에 `options: {"temperature": 0.4}` 적용
(`OLLAMA_TEMPERATURE` 환경변수로 재정의 가능). 0.4는 저확률 토큰 누출을
억제하면서도 답변이 매번 똑같아지지 않는 절충값.

**검증**: 동일 질문("점심 메뉴 하나만 추천해줘") 3회 실행(매회 이력 초기화)
→ 누출 없음 + 다양성 유지 확인.

| 회차 | 응답 |
|---|---|
| 1 | 김치찌개와 밥을 추천합니다. |
| 2 | 김치찌개와 밥을 추천해 드립니다. |
| 3 | 삼겹살 비빔밥입니다. |

## 사후 개선 3. 세션 분리 (v2 로드맵 1번 구현)

**배경**: 전역 이력 하나를 모든 클라이언트가 공유해, 테스트용 curl 대화가
브라우저 대화에 섞여 들어오는 문제를 실제로 겪음(사용자가 "안녕?"에 경복궁
얘기가 나온다고 보고). v2 로드맵 1번을 앞당겨 구현.

**구현 방식**
- Flask `session` 쿠키에 **세션 ID(uuid4)만** 저장하고, 이력 본문은 서버
  메모리의 `histories: dict[세션ID, 이력]`에 유지 (쿠키 4KB 제한 회피).
- `app.secret_key`는 `FLASK_SECRET_KEY` 환경변수, 미지정 시 프로세스마다
  랜덤 생성(재시작하면 쿠키 무효화 — 로컬 용도로 충분).
- `/chat`·`/reset` 모두 현재 세션의 이력만 조작. 성공 후에만 이력을 확정하는
  기존 무결성 패턴은 세션별 리스트에 그대로 적용.

**검증** (curl 쿠키 저장소 2개로 사용자 A/B 시뮬레이션)

| # | 시나리오 | 기대 | 결과 |
|---|---|---|---|
| 1 | A: "제 이름은 승현입니다" | 인사 | ✅ "안녕하세요, 승현님!" |
| 2 | B: "제 이름 알아?" | 모름 | ✅ "모르겠습니다." (A의 대화 안 섞임) |
| 3 | A: "제 이름은?" | 기억 | ✅ "승현입니다." |
| 4 | A만 `/reset` 후 A: "제 이름은?" | 모름 | ✅ "모르겠습니다." |
| 5 | B 이력 길이 (A reset의 영향) | 불변(4) | ✅ history_length 4 |

**남은 한계**: 이력이 프로세스 메모리에만 있어 서버 재시작 시 소멸하고,
오래된 세션이 계속 쌓이면 메모리가 증가한다(만료 정리 없음). 영구 저장·만료는
v2 로드맵의 DB 과제로 넘김.

## 사후 개선 4. SSE 응답 스트리밍 (v2 로드맵 4번 구현)

**배경**: `stream: false`에서는 생성이 끝날 때까지(qwen2.5 기준 3~6초) 사용자가
"…"만 보게 됨. Ollama가 NDJSON 청크를 준다는 건 STEP 2에서 확인했으므로,
이를 모으지 않고 브라우저까지 중계하도록 변경.

**구현 방식**
- `/chat`이 `text/event-stream`(SSE)으로 응답. Python 제너레이터가 Ollama의
  NDJSON 청크를 받는 즉시 `data: {"delta": "..."}` 이벤트로 `yield`.
- 검증 오류(400)·연결 불가(503/504)·Ollama 오류(502)는 스트림 시작 전에
  기존처럼 **JSON으로** 반환 — 프론트는 `r.ok`로 분기.
- 스트림 도중 오류는 `data: {"error": ...}` 이벤트로 전달.
- **이력 무결성 유지**: 스트림이 `done: true`까지 완주한 경우에만 세션 이력에
  user/assistant 쌍을 확정. 중간에 끊기면 이력 불변.
- Flask 세션은 요청 컨텍스트 밖(제너레이터 실행 중)에서 접근 불가하므로,
  이력 리스트 참조를 제너레이터 클로저에 미리 캡처해 전달.
- 프론트: `fetch` + `ReadableStream`으로 `data: ...\n\n` 단위 파싱, 델타를
  말풍선에 실시간 이어 붙임.

**검증** (scratchpad/test_stream.py)

| # | 항목 | 결과 |
|---|---|---|
| 1 | 스트리밍 동작 | ✅ 델타 62개 수신, **첫 델타 0.25s** (전체 2.88s) |
| 2 | 완주 후 이력 확정 + 2턴 문맥 | ✅ "승현" 기억, history_length 4 |
| 3 | 미설치 모델 → JSON 502 | ✅ |
| 4 | 잘못된 JSON → 400 | ✅ |
| 5 | `/reset` | ✅ |

**효과**: 총 생성 시간은 동일하지만 첫 글자 표시가 ~3초 → **0.25초**로 단축.

### 트러블슈팅: 배포 직후 "채팅 답이 안 나옴" 버그

- **증상**: 브라우저에서 메시지를 보내면 답 없이 페이지만 리로드됨.
- **진단 과정**: Flask 로그를 보니 새로고침(GET /)만 찍히고 **POST /chat이
  아예 없음** → 요청이 서버에 도달하기 전에 죽는다는 뜻. 서빙된 페이지에서
  `<script>` 블록을 추출해 `node --check`로 검사하니 문법 오류 검출.
- **원인**: 프론트 JS의 SSE 구분자 `"\n\n"`(및 주석 속 `\n\n`)을 파이썬이
  일반 문자열 리터럴 안에서 **실제 줄바꿈으로 이스케이프 처리**, JS 문자열이
  중간에 끊겨 스크립트 전체가 파싱 실패 → submit 핸들러 미등록 → 폼이 기본
  동작(GET 제출)으로 페이지 리로드.
- **해결**: 템플릿을 raw 문자열(`PAGE = r"""..."""`)로 변경. `node --check`
  통과 및 스트리밍 테스트 5종 재통과 확인.
- **교훈**: 파이썬 문자열 안에 JS를 내장할 때 `\n` 같은 이스케이프가 이중
  해석된다. 내장 템플릿은 raw 문자열이 안전하고, API 테스트(`test_stream.py`)만
  통과했다고 프론트가 멀쩡한 건 아니므로 **서빙된 HTML/JS 자체를 검증**하는
  단계가 필요하다.

## 사후 개선 5. 2차 모델 비교 — 기본 모델을 exaone3.5로 교체

**배경**: qwen2.5의 남은 한계(간헐적 영어 누출, 사실 오류) 개선의 1단계로

qwen3:8b는 사고 모드가 기본이라 공정 비교를 위해 `think: false` 적용.

| exaone3.5 (7.8B, 한국어 특화) | c7c4e3d1ca22 | 4.8 GB | 6.6s | 4.2s | 25.1 |
| qwen3:8b (think off) | 500a1f067a9f | 5.2 GB | 4.3s | 1.6s | 23.6 |

## 사후 개선 6. RAG 구현 (v2 로드맵 5번) — 사실 오류의 구조적 해결

**구성**: `data/seoul_attractions.txt`(서울 명소 10곳, 검증된 사실만 수록) →
`rag.py`(임베딩·코사인 검색, 순수 파이썬) → `app.py` 통합(시스템 프롬프트에
근거 주입 + UI에 "📄 참고 자료" 표시).

**임베딩 모델 선정 과정** (검색 품질 실측 비교):

| 모델 | "경복궁 어디?" 1위 | 무관 질문("날씨") 필터링 |
|---|---|---|
| nomic-embed-text (접두사 없음) | ✕ 한강공원 | ✕ 0.736으로 통과 |
| nomic-embed-text (`search_query:` 접두사) | ✓ 0.763 | ✕ 0.683 (변별력 부족) |
| **bge-m3** | ✓ 0.774 | **✓ 0.5 미만으로 제외** |

- nomic-embed-text는 모델 카드가 요구하는 용도 접두사를 붙여야 성능이 나오며,
  붙여도 한국어 변별력이 약했다. 다국어 특화 **bge-m3**(1.2GB)로 교체하니
  정답 1위 + 무관 질의 필터링이 모두 해결돼 기본 임베딩 모델로 채택.

**동작 방식**: 질문마다 유사도 상위 3개 청크 중 0.5 이상만 시스템 프롬프트에
"자료에 근거해 답하라"와 함께 주입. 관련 자료가 없으면 주입하지 않아 일반
대화는 그대로 동작. 임베딩 호출 실패 시에도 RAG 없이 계속 동작(우아한 강등).

**검증** (이전에 실제로 환각이 났던 질문 포함):

| 질문 | 이전 (환각) | RAG 적용 후 |
|---|---|---|
| 경복궁은 어느 구에? 언제 지어졌어? | "용산구, 국립박물관" ✕ | **"종로구, 1395년"** ✓ (참고: 경복궁) |
| 서울 최고층 건물? 높이? | - | "롯데월드타워, 약 555m" ✓ |
| N서울타워 전망대 개방 시기? | - | "1980년부터" ✓ |
| 파이썬이 뭐야? (자료 무관) | - | 참고 자료 없이 일반 답변 ✓ |
